# Introduction

Understanding virtual networking is essential before building any penetration testing lab.

Every machine in your lab must be able to communicate correctly. Choosing the wrong network mode may prevent hosts from seeing each other or accidentally expose vulnerable machines to your home network.

This chapter explains the most common networking modes used in **VirtualBox** and **VMware**.

---

# Why Virtual Networking Matters

During a penetration test, you will typically have:

- An attacker machine (Kali Linux)
- One or more target machines
- A virtual network connecting them

Choosing the correct network mode allows you to simulate real-world environments safely.

Benefits include:

- Isolated practice environments
- Safe exploitation of vulnerable systems
- Internet access when required
- Multiple network segments
- Internal penetration testing scenarios

---

# How Virtual Machines Communicate

Each virtual machine has one or more **virtual network adapters**.

Think of them as physical Ethernet cards.

```
+------------------+
|   Kali Linux     |
|   Adapter 1      |
+---------+--------+
          |
          |
     Virtual Switch
          |
+---------+--------+
| Target Machine   |
| Adapter 1        |
+------------------+
```

The hypervisor (VirtualBox or VMware) decides how these adapters communicate.

---

# NAT Network

**NAT (Network Address Translation)** is the default networking mode.

```
Internet
    |
Host Computer
    |
Virtual NAT
    |
----------------------
|                    |
Kali              Target
```

## Characteristics

- VM has Internet access.
- VM can download packages.
- VM cannot normally be reached from your physical network.
- Safest option for beginners.

## Advantages

- Simple configuration
- Internet access
- Good isolation
- Safe for vulnerable machines

## Disadvantages

- Limited communication from outside
- Incoming connections require port forwarding

## Typical Use Cases

- Updating Kali
- Installing tools
- Downloading exploits
- General Linux usage

---

# Bridged Network

Bridged mode connects the VM directly to the physical network.

```
Router
   |
---------------------
|        |         |
PC     Kali      Laptop
```

The VM behaves like a real computer connected to your router.

## Characteristics

- Receives its own IP address.
- Visible to every device on the LAN.
- Can communicate with other physical devices.

## Advantages

- Realistic networking
- Full LAN communication
- Useful for enterprise labs

## Disadvantages

- Less isolated
- Vulnerable machines may be exposed
- Requires caution

## Typical Use Cases

- Active Directory labs
- Internal network simulations
- Multi-host environments

---

# Host-Only Network

Host-Only creates a completely isolated virtual network.

```
Host PC
   |
-------------------------
|                       |
Kali              Windows VM
```

No Internet access.

Only the host and virtual machines can communicate.

## Characteristics

- Completely isolated
- Safe
- No external access

## Advantages

- Secure
- Ideal for vulnerable machines
- Excellent for malware analysis
- Prevents accidental exposure

## Disadvantages

- No Internet connection
- Updates require another adapter

## Typical Use Cases

- Metasploitable
- OWASP Broken Web Apps
- Windows privilege escalation labs
- eJPT practice

---

# Internal Network

Internal networking is similar to Host-Only but even more isolated.

```
Kali
   |
Internal Switch
   |
Windows
   |
Linux
```

The host computer cannot communicate with these machines.

Only virtual machines connected to the same internal network can communicate.

This is useful for advanced attack simulations.

# When to Use Each Mode

## NAT

Use when:

- Installing Kali
- Updating packages
- Downloading tools
- Internet access is required

---

## Bridged

Use when:

- Practicing enterprise networking
- Building Active Directory labs
- Simulating corporate environments

---

## Host-Only

Use when:

- Practicing exploitation
- Running intentionally vulnerable machines
- Learning privilege escalation
- Studying malware safely

---

## Internal

Use when:

- Building advanced attack chains
- Practicing pivoting
- Simulating segmented corporate networks


# Checking Network Configuration

Display interfaces:

```bash
ip addr
```

Display routing table:

```bash
ip route
```

Display DNS configuration:

```bash
cat /etc/resolv.conf
```

Display active interfaces:

```bash
nmcli device status
```

---

# Testing Connectivity

Check Internet:

```bash
ping 8.8.8.8
```

Check DNS:

```bash
ping google.com
```

Ping another VM:

```bash
ping 192.168.56.101
```

Display listening services:

```bash
ss -tulnp
```

# Troubleshooting

## No Internet

Check:

```bash
ip addr
```

Verify:

```bash
ip route
```

Restart networking:

```bash
sudo systemctl restart NetworkManager
```

---

## Cannot Reach Another VM

Verify:

- Same virtual network
- Correct IP addresses
- Firewall disabled (for testing)
- Adapter enabled

Test with:

```bash
ping TARGET_IP
```

---

## Wrong IP Address

Request a new DHCP lease:

```bash
sudo dhclient
```

# Best Practices

- Never expose intentionally vulnerable machines to your home network.
- Use **Host-Only** whenever possible for practice labs.
- Keep Kali updated using a NAT adapter.
- Use snapshots before changing networking settings.
- Label your virtual adapters clearly.
- Document the IP addresses of every machine.
- Learn to read routing tables and interface configurations.

A well-designed lab is easier to manage, safer to use, and more realistic.
