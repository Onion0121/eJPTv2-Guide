# Introduction

SSH pivoting is a technique that uses the SSH protocol to route traffic through a compromised machine and access internal systems.

SSH is commonly used for administration on Linux systems, but it can also be used by penetration testers to create tunnels and access restricted networks.

The objective:

```
Attacker Machine

↓

SSH Pivot Host

↓

Internal Network

↓

Hidden Target
```

---

# What is SSH Pivoting?

SSH pivoting allows an attacker or penetration tester to use a compromised SSH-accessible host as a gateway.

The pivot machine has access to networks that the attacker cannot directly reach.

Example:

```
Attacker

192.168.1.50


↓

Pivot Server

192.168.1.20

10.10.10.5


↓

Internal Server

10.10.10.25
```

The attacker creates an SSH tunnel through:

```
192.168.1.20
```

to reach:

```
10.10.10.25
```

---

# Why SSH Pivoting Matters

Organizations commonly separate networks:

```
Internet

↓

DMZ

↓

Internal Servers

↓

Database Network
```

A compromised server in the DMZ may have access to internal systems.

SSH pivoting allows testers to:

- Reach hidden services
- Scan internal networks
- Access restricted applications
- Perform lateral movement

SSH pivoting requires:

```bash

```

# Local Port Forwarding

Local forwarding allows the attacker to access an internal service through the SSH pivot.

Syntax:

```bash
ssh -L LOCAL_PORT:TARGET_IP:TARGET_PORT user@PIVOT_IP
```

---

Example:

Internal web server:

```
10.10.10.20:80
```

Create tunnel:

```bash
ssh -L 8080:10.10.10.20:80 user@192.168.1.20
```

Now access:

```
http://127.0.0.1:8080
```

Traffic:

```
Browser

↓

localhost:8080

↓

SSH Tunnel

↓

Pivot

↓

10.10.10.20:80
```

---

# Remote Port Forwarding

Remote forwarding allows a remote machine to access a service through the SSH connection.

Syntax:

```bash
ssh -R REMOTE_PORT:LOCAL_IP:LOCAL_PORT user@PIVOT_IP
```

Example:

```bash
ssh -R 9000:127.0.0.1:9000 user@pivot
```

Useful when:

- The attacker needs to expose a local service
- The pivot cannot directly connect back

---

# Dynamic Port Forwarding

Dynamic forwarding creates a SOCKS proxy.

Syntax:

```bash
ssh -D PORT user@PIVOT_IP
```

Example:

```bash
ssh -D 9050 user@192.168.1.20
```

Creates:

```
SOCKS5 Proxy

127.0.0.1:9050
```

Any compatible application can route traffic through the pivot.

---

# Proxychains Configuration

Proxychains routes applications through a proxy.

Edit configuration:

```bash
nano /etc/proxychains.conf
```

Add:

```
socks5 127.0.0.1 9050
```

Run tools:

```bash
proxychains nmap -sT 10.10.10.0/24
```

---

Example:

Without proxy:

```
Kali

X

Internal Network
```

With proxy:

```
Kali

↓

SSH SOCKS

↓

Pivot

↓

Internal Network
```

# SSH Pivoting With Multiple Networks

Sometimes multiple pivots are required.

Example:

```
Attacker

↓

Pivot 1

↓

Pivot 2

↓

Internal Server
```

Each SSH connection extends access deeper into the environment.

---

Example:

First tunnel:

```bash
ssh -D 9050 user@pivot1
```

Second tunnel:

```bash
ssh -D 9051 user@pivot2
```

---

# SSHuttle Alternative

SSHuttle creates a VPN-like connection through SSH.

Example:

```bash
sshuttle -r user@pivot 10.10.10.0/24
```

Advantages:

- Automatic routing
- No proxychains required
- Easier internal access

# Troubleshooting SSH Tunnels

## Connection Refused

Check:

```bash
ping TARGET
```

Verify:

- Target is reachable
- Port is open
- SSH permissions

---

## Tunnel Created But Service Unreachable

Check:

```bash
nmap TARGET
```

Verify:

- Correct IP
- Correct port
- Firewall rules

---

## Proxychains Not Working

Check:

Configuration:

```bash
cat /etc/proxychains.conf
```

Verify:

```
socks5 127.0.0.1 9050
```

# Defense & Mitigation

Prevent SSH pivoting by:

- Restricting SSH access
- Using key-based authentication
- Monitoring SSH sessions
- Limiting user permissions
- Segmenting networks
- Restricting forwarding features

SSH configuration:

```
AllowTcpForwarding no
```

can restrict tunneling.
